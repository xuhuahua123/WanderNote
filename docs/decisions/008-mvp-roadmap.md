# 008. MVP 开发路线图与多 Agent 分工

## 状态

已确认。

## 背景

本项目后续会交给不同开发 Agent 分别实现：

- 前端 Agent：负责 H5 前端
- 后端 Agent：负责 Spring Boot 后端
- Photo-Agent Agent：负责家庭电脑照片同步程序

因此开发路线必须明确边界、接口依赖和验收标准，避免各自实现后无法联调。

## 总体原则

```text
先接口，后页面。
先索引，后图片。
先路线，后写作。
先跑通主链路，再优化体验。
```

三个开发 Agent 都必须以以下文档为准：

- `001-frontend-mvp.md`
- `002-photo-agent.md`
- `003-backend-data-model.md`
- `004-ai-writing-service.md`
- `005-api-contract.md`
- `006-external-services-and-secrets.md`
- `007-project-architecture.md`

任何实现发现文档冲突，先回到主线程讨论，不允许自行扩展大功能。

## 角色分工

### 前端 Agent

负责：

- Vue3 H5 工程
- 访问码登录页
- 首页 / 路线列表
- 路线详情页
- 添加城市页
- 添加地点页
- 照片选择弹窗
- 写整条路线选图弹窗
- AI 写作准备页
- 文章结果页
- 同步状态页
- 自动保存前端逻辑
- 图片 URL 过期后的刷新逻辑

不负责：

- 腾讯地图直接调用
- COS 直接签名
- MiMo 直接调用
- Photo-Agent 同步逻辑
- 数据库结构

### 后端 Agent

负责：

- Spring Boot 工程
- MySQL 数据模型
- Redis 心跳状态
- 访问码登录
- 行政区划初始化
- 路线/城市/地点接口
- 腾讯地图调用和 POI 缓存
- 照片 GPS 匹配
- COS 上传/下载签名
- Photo-Agent 接口
- AI 写作服务
- 草稿保存
- 同步状态接口
- 错误码和鉴权

不负责：

- 前端页面实现
- Photo-Agent 本地扫描
- 本地图片压缩
- 桌面 GUI

### Photo-Agent Agent

负责：

- Python Agent 工程
- 本地配置文件
- 本地 SQLite
- 扫描照片目录
- 读取 EXIF 时间和 GPS
- 生成 thumb / preview
- 批量上传照片索引
- 申请 COS 临时上传 URL
- 直传 COS
- 通知后端上传完成
- 心跳上报
- 上传限速和失败重试
- 本地日志

不负责：

- 路线管理
- 地点解析
- 腾讯地图调用
- AI 写作
- 前端界面
- COS 永久密钥管理

## 开发阶段

## M0. 工程骨架

目标：

建立清晰项目结构。

交付：

```text
frontend/
backend/
photo-agent/
deploy/
scripts/
docs/
```

前端：

- Vue3 + Vite + TypeScript + Vant 初始化。
- 基础路由和页面壳。

后端：

- Spring Boot 初始化。
- MySQL / Redis 配置。
- 统一响应结构。
- 统一错误处理。

Photo-Agent：

- Python 项目初始化。
- 配置文件读取。
- SQLite 初始化。
- 日志初始化。

验收：

- 三个工程可独立启动。
- README 写清启动方式。
- `.env.example` / `config.example.yaml` 存在。
- 无真实密钥提交。

## M1. 后端基础与行政区划

目标：

后端具备基础登录、行政区划和健康检查能力。

后端交付：

- 访问码登录
- token 续期
- 省份列表
- 城市/地区列表
- 腾讯行政区划初始化脚本
- Redis 可用性检查
- MySQL migration

前端交付：

- 访问码登录页接入真实接口
- 首页空状态
- 新建路线基础 UI

Photo-Agent：

- 暂无强依赖，可继续本地扫描模块

验收：

- H5 能通过访问码登录。
- 后端能拉取并保存腾讯行政区划。
- 前端能查询省份和城市列表。

## M2. Photo-Agent 索引与 COS 上传

目标：

打通本地照片到云端索引和 COS 图片资源。

后端交付：

- Agent 鉴权
- 心跳接口
- 批量照片索引上传
- COS 上传地址申请
- COS 上传完成通知
- 图片短期下载 URL 生成
- 同步状态接口

Photo-Agent 交付：

- 扫描照片目录
- 读取 EXIF 时间/GPS
- 生成 thumb / preview
- 上传照片索引
- 申请 COS 上传地址
- 直传 COS
- 上传完成通知
- 30 秒心跳
- 本地 SQLite 状态

前端交付：

- 同步状态页接入真实接口

验收：

- Agent 能扫描测试照片。
- 后端能保存 photo / photo_asset。
- COS 能看到 thumb / preview 对象。
- H5 同步状态页能显示：
  - Agent 在线
  - 最近同步
  - 照片总数
  - GPS 照片数量
  - 缩略图/预览图同步数量

## M3. 路线、城市、地点和照片匹配

目标：

打通路线管理和地点自动匹配照片。

后端交付：

- 路线列表
- 新建路线
- 路线详情
- 添加城市
- 添加地点
- 腾讯地点 suggestion 调用
- POI 缓存
- 城市上下文过滤
- GPS 照片匹配
- 地点照片分页
- 移除误匹配照片
- 换一批地点照片
- 写整条路线代表照片推荐

前端交付：

- 首页 / 路线列表
- 新建路线
- 路线详情页
- 添加城市页
- 添加地点页
- 照片选择弹窗
- 写整条路线选图弹窗

Photo-Agent：

- 根据联调问题补充同步稳定性

验收：

- 用户能新建路线。
- 用户能添加城市。
- 用户能在城市下输入地点。
- 后端能解析地点坐标。

- 可以换一批。

## M4. AI 写作与草稿

目标：

打通从选图到生成文章、编辑和保存。

后端交付：

- AI 写作生成接口
- 重新生成接口
- 更朴实一点
- 更有感情一点
- article_draft 保存/读取
- article_generation_log
- MiMo-V2.5 调用封装

前端交付：

- AI 写作准备页
- 文章结果页
- 标题/正文/图片说明编辑
- 自动保存
- 复制全文
- 重新生成二次确认
- 更朴实/更有感情按钮

Photo-Agent：

- 无新增主功能

验收：

- 写某个地点能生成文章。
- 写整条路线能生成文章。
- 图片说明能生成。
- 编辑后能自动保存。
- 返回后点击继续写能恢复。
- 复制全文可用。
- AI 调用日志可查询。

## M5. 联调、异常处理和体验修正

目标：

让 MVP 对真实用户可用。

重点：

- 登录过期不丢文章
- 图片签名 URL 过期可刷新
- Agent 离线状态准确
- COS 上传失败可重试
- 腾讯地图无结果时可保存地点
- AI 失败时提示重试
- 2C2G 服务器内存稳定
- 前端移动端样式检查

验收：

- 真实手机可完整跑通：
  1. 登录
  2. 新建路线
  3. 添加城市
  4. 添加地点
  5. 自动匹配照片
  6. 选择照片
  7. 生成游记
  8. 编辑文章
  9. 复制全文
  10. 查看同步状态

## M6. MVP 验收

验收标准：

前端：

- 字体大
- 按钮清楚
- 页面层级少
- 无独立草稿箱
- 首页和路线详情符合原型 v2 方向

后端：

- 数据模型稳定
- 接口符合 `005-api-contract.md`
- 外部密钥不泄露
- Redis 心跳可用
- COS 私有读写和签名 URL 可用

Photo-Agent：

- 能稳定扫描 1 万张照片
- 能增量扫描
- 能生成 thumb / preview
- 能限速上传
- 能失败重试
- 不开放本地端口

业务：

- 原图不上云
- thumb / preview 存 COS
- 地点能用 GPS 自动匹配照片
- 一篇文章最多 10 张照片
- AI 能生成公众号游记

## 多 Agent 协作规则

1. 后端 Agent 先实现接口 mock 或最小真实接口。
2. 前端 Agent 优先按 `005-api-contract.md` 调接口，不自行改字段。
3. Photo-Agent Agent 优先跑通测试照片目录，不依赖完整前端。
4. 字段变更必须先改接口文档，再改代码。
5. 接口不确定时，不允许各 Agent 自行猜。
6. 每个 Agent 完成阶段任务后，需要提供：
   - 改动摘要
   - 启动方式
   - 测试方式
   - 已知问题
7. 代码 review 重点检查：
   - 是否越界实现
   - 是否泄露密钥
   - 是否破坏文档约定
   - 是否能在 2C2G 约束下运行

## 推荐任务派发顺序

第一批：

```text
后端 Agent：M0 + M1
Photo-Agent Agent：M0 + 本地扫描原型
前端 Agent：M0 + 静态页面原型
```

第二批：

```text
后端 Agent：M2 接口
Photo-Agent Agent：M2 联调
前端 Agent：同步状态页
```

第三批：

```text
后端 Agent：M3 路线和匹配
前端 Agent：M3 页面接入
Photo-Agent Agent：同步稳定性
```

第四批：

```text
后端 Agent：M4 AI 写作
前端 Agent：M4 写作页面
```

第五批：

```text
全体：M5/M6 联调验收
```

## 产品原则

MVP 目标不是功能多，而是主链路稳定：

```text
照片同步 -> 路线搭建 -> 自动匹配照片 -> 生成游记 -> 复制公众号
```

三个开发 Agent 都围绕这条主链路交付。
