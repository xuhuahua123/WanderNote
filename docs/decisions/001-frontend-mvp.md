# 001. 前端 MVP 功能决策记录

## 状态

已确认。

## 背景

本项目是面向退休用户的手机端旅行公众号写作助手。

用户不会按“电脑文件夹”或“行政区划层级”来理解旅行照片，而是会按一趟旅行路线来回忆：

```text
从哪里出发 -> 经过哪些城市 -> 去了哪些地方 -> 发生了什么故事
```

因此，路线功能既要让前端操作足够简单，也要给后端足够清晰的地点上下文，方便通过照片 GPS 自动匹配照片。

本文件记录当前已经确认的前端 MVP 功能范围和关键交互决策。

它不是最终 PRD，也不是接口文档。后续完整 PRD 会在所有模块讨论完成后再统一整理。

当前文档覆盖：

- 首页 / 路线列表
- 路线详情页
- 添加城市 / 添加地点
- 地点照片选择
- 写整条路线
- AI 写作准备页
- 文章结果页
- 草稿保存方式
- 登录 / 访问码
- Agent 同步状态页
- 与前端相关的后端匹配边界

## 整体路线结构

路线采用两级结构：

- 省份只做视觉分组，不作为主要操作节点。
- 市/地区作为一级路线节点，必须从腾讯地图行政区划数据中选择，不能自由输入。
- 具体地点作为二级路线节点，可以由用户自由输入，也可以从腾讯地图 POI 候选中选择。

示例：

```text
2025 西藏新疆大环线

河南省
  郑州市
    出发

陕西省
  西安市
    西安城墙

西藏自治区
  拉萨市
    布达拉宫
  日喀则市
    珠峰北坡
    珠峰大本营

新疆维吾尔自治区
  喀什地区
    喀什古城
```

## 路线详情页 / 添加城市和地点

路线详情页按旅行顺序展示城市。

省份根据城市所属关系自动显示为小标题和颜色分组，不需要用户手动维护。

路线详情页定位为：

```text
一趟旅行的目录页 + 写文章入口
```

它不是地图页、相册管理页或复杂编辑器。用户打开后，应能马上看到这趟旅行经过哪些城市、每个地点有多少照片，以及从哪里开始写文章。

页面不做地图轨迹、拖拽排序、多选管理、照片删除、原图下载、复杂筛选或富文本编辑。

## 地点照片页 / 照片选择

地点下匹配到的照片和写文章时选择的照片不是一回事。

- 地点可以匹配到很多照片。
- 写一篇文章最多选择 10 张照片。
- 写文章时只使用用户当前选中的照片。

MVP 不做复杂独立相册管理页，采用弹窗或底部抽屉展示地点照片。

用户点击地点或点击写文章前，打开照片选择弹窗：

```text
珠峰北坡照片

已选择 4 / 10

[照片] [照片] [照片]
[照片] [照片] [照片]

下拉继续加载更多照片

[ 取消 ]        [ 用这些照片写文章 ]
```

照片选择规则：

- 照片以缩略图网格展示。
- 支持下拉分页加载，避免一次加载过多照片。
- 最多选择 10 张。
- 已选照片要有明显标记。
- 误匹配或不想使用的照片可以移除。
- 点击照片可以预览大图。
- 预览页要能返回选择弹窗。
- 不做原图下载。
- 不做照片删除。
- 不修改本地照片文件。

按钮和状态必须清晰直白，避免只用图标表达关键操作。

推荐按钮文案：

```text
看大图
移除
选择
取消选择
换一批
用这些照片写文章
取消
```

老人端不能出现含糊按钮，例如只显示一个图标但没有文字说明。

`换一批`作为通用选图能力：

- 地点写作可以换一批推荐照片。
- 整条路线写作可以换一批代表照片。
- 用户不需要理解推荐规则，只看到新的照片组合。
- 换一批后仍然最多选择 10 张。
- 用户可以继续移除不想用的照片。

## 写整条路线

写整条路线不要求用户从大量照片中手动挑选。

整条路线文章采用：

```text
系统自动推荐代表照片 -> 用户确认 -> 进入写作
```

入口位于路线详情页顶部：

```text
[ 写整条路线 ]
```

点击后打开代表照片确认弹窗：

```text
写整条路线

系统已从 428 张照片里选出 10 张代表照片

[照片] [照片] [照片]
[照片] [照片] [照片]

[ 换一批 ]    [ 用这些照片写游记 ]
```

整条路线最多使用 10 张照片。

推荐照片来源：

1. 出发地点 1 张。
2. 终点地点 1 张。
3. 每个省份尽量至少 1 张，如果该省有匹配照片。
4. 重要地点按路线顺序补足。
5. 总数不超过 10 张。

MVP 的后端推荐规则可以先保持简单：

- 每个地点取 1 张代表照片。
- 优先按路线顺序取。
- 优先取有 GPS 且距离地点坐标较近的照片。
- 超过 10 张时截取 10 张。
- 用户点“换一批”时，后端避开上一批照片，重新生成一组候选。

起点和终点也作为路线里的普通地点处理，而不是单独做一套结构。

示例：

```text
河南省
  郑州市
    出发
    回到家
```

这样写整条路线时，系统可以自然地把出发和返程照片放入文章首尾。

## AI 写作准备页

AI 写作准备页只保留少量输入，不做复杂 prompt 表单。

页面只包含三块：

1. 当前写作对象和已选照片数量。
2. 情绪/文风选择。
3. 一段补充说明，可不填。

重点内容、用户备注、个人感受合并到同一个输入框里。

页面示例：

```text
写游记

珠峰北坡
已选择 8 张照片

这篇想写成什么感觉？

[ 轻松自在 ] [ 一路辛苦但值得 ]
[ 震撼感动 ] [ 温暖踏实 ]
[ 怀念回味 ] [ 朴实记录 ]

这一路/这一站想重点写什么？
可以写玩了什么、路上发生了什么、吃住怎么解决、看到什么风景、心里有什么感受。

[ 那天风很大，第一次这么近看到珠峰，心里很震撼。 ]

[ 生成游记 ]
```

情绪选项：

- 轻松自在
- 一路辛苦但值得
- 震撼感动
- 温暖踏实
- 怀念回味
- 朴实记录

情绪设计要贴合退休自驾旅行场景：

- 退休后时间自由，可以慢慢走。
- 自己开车走长线，有成就感。
- 很多时间在车上，吃住靠自己准备。
- 路上辛苦，但准备充分，心里踏实。
- 家人或同行者互相照应。
- 看到雪山、草原、边疆、古城时有震撼和感动。

补充说明允许为空。

如果用户什么都不写，系统仍然可以根据路线、城市、地点、已选照片、照片时间和情绪选项生成文章。

按钮文案可以根据是否填写说明轻微变化：

```text
生成游记
直接生成游记
```

不做：

- prompt 输入框
- 字数滑块
- 复杂模板选择
- 多个备注框
- 必填的大段文字
- 专业写作参数

写地点时，文章围绕当前地点和照片展开。

写整条路线时，文章按路线顺序展开，并在结尾加入退休后长途自驾的回忆和感悟。

## 文章结果页

文章结果页采用分段式简易编辑器。

目标是让用户能修改标题、正文和图片说明，并复制到公众号后台。

不做复杂富文本排版器。

推荐展示结构：

```text
文章已生成

标题
[ 退休后第一次近看珠峰，心里满是震撼 ]

正文
[
这一次来到珠峰北坡，是我们退休后自驾路上最难忘的一站……
]

图片说明

照片 1
[ 蓝天下的珠峰，远远望去格外清晰 ]

照片 2
[ 我们在车边准备简单的午饭 ]

[ 复制全文 ]

[ 更朴实一点 ] [ 更有感情一点 ]
[ 重新生成 ]

已自动保存草稿
```

必须支持：

- 标题编辑
- 正文编辑
- 每张图片说明编辑
- 复制全文
- 自动保存草稿
- 重新生成
- 更朴实一点
- 更有感情一点

复制全文格式建议：

```text
标题

正文第一段

【图片1】
图片说明：xxxx

正文第二段

【图片2】
图片说明：xxxx

结尾
```

复制成功后提示：

```text
已复制，可以去公众号粘贴了
```

重新生成规则：

- 保留当前照片。
- 保留当前情绪选项。
- 保留用户补充说明。
- 重新生成一版新文章。
- 当前文章应自动保存为草稿，避免被覆盖。

快速改写规则：

- `更朴实一点`：基于当前文章调整文风，让语言更像生活记录，减少夸张表达。
- `更有感情一点`：基于当前文章调整文风，增加回忆、感受和旅途情绪。
- 不重新选图。
- 不重新进入写作准备页。

暂不做：

- 字体选择
- 颜色设置
- 加粗/斜体工具栏
- 图片拖拽排序
- 图片上传公众号
- 一键发布公众号
- 复杂版本对比
- 多栏模板

公众号后台本身负责最终排版，本系统只负责生成内容、轻量编辑和复制。

## 草稿管理

MVP 不做独立草稿中心。

草稿不是一个单独功能，而是“写”功能的保存状态。

用户在哪里写，草稿就挂在哪里。

地点写作：

```text
珠峰北坡
23 张照片              [写]
```

如果该地点已经生成或编辑过文章：

```text
珠峰北坡
23 张照片              [继续写]
```

整条路线写作同理：

```text
[ 写整条路线 ]
```

如果整条路线已经生成或编辑过文章：

```text
[ 继续写整条路线 ]
```

不做：

- 我的草稿
- 草稿历史
- 多版本列表
- 草稿搜索
- 草稿分类
- 草稿管理中心

仍然必须自动保存：

- 文章生成后自动保存。
- 用户修改标题、正文、图片说明后自动保存。
- 用户下次点击 `[继续写]` 时，回到上次编辑结果。

重新生成规则调整为：

- 因为不做历史草稿，重新生成会替换当前文章。
- 点击重新生成时需要二次确认。

提示文案：

```text
重新生成会替换当前文章，是否继续？

[ 取消 ] [ 重新生成 ]
```

快速改写也会替换当前文章，但完成后自动保存。

后端只需要保存每个写作对象的当前草稿：

```text
article_draft
- id
- target_type: route / place
- target_id
- title
- content
- image_captions
- selected_photo_ids
- mood
- user_note
- updated_at
```

每个路线或地点只保留一份当前草稿。

## 首页 / 路线列表

首页定位为：

```text
我的旅行路线首页
```

用户打开系统后，第一屏应能清楚看到已有路线，并能快速新建路线、搜索路线、继续写文章。

页面示例：

```text
我的旅行

[ 搜索路线、城市、地点 ]

[ + 新建路线 ]

[封面图]
2025 西藏新疆大环线
河南出发 · 西藏 · 新疆 · 内蒙古 · 回到河南
12 城市 · 36 地点 · 428 照片
最近写过：珠峰北坡
[ 查看路线 ] [ 继续写 ]

[封面图]
2025 春天洛阳赏花
洛阳 · 2 城市 · 5 地点 · 68 照片
[ 查看路线 ]
```

路线卡片建议包含：

- 封面图
- 路线名称
- 年份
- 简短路线摘要
- 城市数量
- 地点数量
- 照片数量
- 最近写作状态
- 查看路线按钮
- 继续写按钮，如果存在当前草稿

封面图由系统自动从路线照片中选择，不要求用户手动上传。

新建路线入口必须明显：

```text
[ + 新建路线 ]
```

空状态：

```text
还没有旅行路线

可以先新建一趟旅行，
再按城市和地点整理照片。

[ 新建第一条路线 ]
```

新建路线表单 MVP 保持简单：

```text
新建路线

路线名称
[ 2025 西藏新疆大环线 ]

年份
[ 2025 ]

[ 保存 ]
```

起点和终点不放在新建路线表单里，统一作为路线里的地点添加。

搜索必须支持，但保持轻量。

搜索范围：

- 路线名称
- 城市名
- 地点名

搜索结果可以直接过滤首页路线卡片，并显示匹配提示。

示例：

```text
搜索：珠峰

2025 西藏新疆大环线
匹配：日喀则市 / 珠峰北坡
```

首页暂不做：

- 路线删除
- 路线归档
- 路线排序
- 路线分组
- 多用户切换
- 复杂筛选

首页视觉方向：

- 大卡片
- 大字体
- 封面图明显
- 信息少而清楚
- 主按钮明确
- 有私人旅行相册的温暖感
- 不做营销页式 banner
- 不堆复杂操作

## 登录 / 访问码

MVP 使用访问码登录，不做账号密码体系。

不做：

- 注册
- 找回密码
- 多用户切换
- 复杂权限系统

登录页示例：

```text
退休旅行写作助手

把一路风景，
写成自己的游记。

请输入访问码

[        ]

[ 进入 ]
```

登录页视觉要求：

- 要做得好看、温暖、清楚。
- 不要像后台管理系统登录框。
- 字体要大。
- 输入框要大。
- 按钮要明显。
- 可以使用一张旅行封面图或柔和背景。
- 不堆复杂说明。

登录有效期规则：

```text
基础有效期：2 小时
活跃自动续期：需要
最长保持：7 天
```

如果用户一直在使用，前端应自动续期，不能因为固定 2 小时到期就打断写作。

写作页必须自动保存：

- 标题修改自动保存。
- 正文修改自动保存。
- 图片说明修改自动保存。
- 停止输入约 2 秒后保存。
- 离开页面前尝试保存一次。

保存状态必须显示：

```text
正在保存...
已自动保存
保存失败，请检查网络
```

登录快过期或续期失败时，不允许直接丢弃当前文章。

提示文案：

```text
登录快过期了，请重新输入访问码。
文章内容已保存在本机，不会丢失。
```

前端必须做本地临时保存：

- 当前编辑标题
- 当前编辑正文
- 当前图片说明
- 当前选中照片
- 当前写作对象

本地缓存可使用 `localStorage` 或 IndexedDB。后端保存成功后再标记同步完成。

核心原则：

```text
登录过期不能导致文章内容丢失。
写作内容必须先保住，再处理登录状态。
```

## 设置 / Agent 同步状态

MVP 不做复杂设置中心。

只做一个清晰的同步状态页，入口放在首页右上角。

入口文案必须明确：

```text
[ 同步状态 ]
```

不要只使用齿轮图标。

同步状态页定位为：

```text
系统体检页
```

如果出现问题，用户可以截图发给家人远程排查。

页面示例：

```text
同步状态

Photo-Agent 在线

最近同步
今天 18:20

照片总数
428 张

有定位照片
421 张

定位覆盖率
98%

缩略图
428 / 428 已同步

预览图
428 / 428 已同步

[ 刷新状态 ]

详细信息 v
```

必须显示：

- Photo-Agent 在线状态
- 最近同步时间
- 照片总数
- GPS 照片数量
- GPS 覆盖率
- 缩略图同步数量
- 预览图同步数量
- 刷新状态按钮

状态文案要直接：

```text
Photo-Agent 在线
Photo-Agent 离线
正在同步照片
```

状态颜色：

- 在线：绿色
- 离线：红色
- 同步中：蓝色或橙色

Agent 离线时显示：

```text
Photo-Agent 离线

家里电脑上的照片同步程序暂时没有连接。
如果刚打开电脑，请等一会儿再刷新。

[ 刷新状态 ]
```

如果 GPS 覆盖率较低，例如低于 80%，显示提醒：

```text
有些照片没有位置信息，可能无法自动匹配到地点。
```

详细信息默认折叠，供远程排查使用：

```text
详细信息

Agent ID：home-mac-001
Agent 版本：0.1.0
最近心跳：2026-05-28 18:20:12
最近扫描：2026-05-28 18:18:02
同步失败：0 张
待上传缩略图：0 张
```

MVP 只做 `[ 刷新状态 ]`。

暂不做：

- 立即同步照片
- Agent 配置修改
- 本地目录配置
- 日志下载
- 复杂设置项

添加一级路线节点：

```text
添加城市

选择省份：
[ 西藏自治区 ]

选择城市/地区：
[ 日喀则市 ]

[ 保存 ]
```

添加城市规则：

- 从路线详情页点击 `[ + 添加城市 ]` 进入。
- 必须先选择省份，再选择城市/地区。
- 城市/地区必须来自腾讯地图行政区划库，不允许自由输入。
- 保存后默认追加到路线最后。
- MVP 不做拖拽排序；如果后续需要调整顺序，再考虑“上移/下移”按钮。

添加二级地点：

```text
日喀则市

添加地点

地点名称：
[ 珠峰北坡 ]

系统自动查找附近照片

[ 保存 ]
```

添加地点规则：

- 从某个城市下面点击 `[ + 添加地点 ]` 进入。
- 地点名称允许用户自由输入。
- 前端不要求用户选择日期、文件夹、坐标或匹配规则。
- 用户输入地点后，系统自动调用后端进行地点解析和照片匹配。
- 保存后回到路线详情页，并显示地点名称和匹配到的照片数量。

添加地点有三种结果状态：

```text
状态 1：找到照片

找到 23 张照片
[缩略图] [缩略图] [缩略图]

[ 保存 ]
```

```text
状态 2：找到地点，但暂时没有照片

已找到这个地点
暂时没有匹配到照片

[ 先保存 ]
```

```text
状态 3：没有找到地点

没有找到这个地点

[ 先保存 ]
```

以上三种情况都允许保存，不能阻塞用户继续搭建路线。

如果后端对地点候选非常不确定，后续版本可以增加极简确认：

```text
你说的是这个地方吗？

珠峰大本营
西藏自治区日喀则市定日县

[ 是的 ] [ 换一个 ]
```

但这不是 MVP 主流程。

正常流程里，不要求用户选择文件夹、日期、坐标或匹配规则。

## 后端匹配规则

当用户在某个城市下面添加地点时：

1. 前端提交 `routeCityId` 和 `placeName`。
2. 后端读取当前城市上下文，包括城市名、省份名、行政区划代码。
3. 后端调用腾讯地图地点提示或 POI 搜索接口。
4. 候选地点必须优先限定在当前城市或当前省份内。
5. 后端拿每个候选地点的坐标去照片 GPS 索引里试匹配。
6. 最终选择综合得分最高的候选地点：
   - 属于当前城市或当前省份
   - POI 类型合理，比如景区、地标、道路、营地等
   - 附近能匹配到照片
   - 匹配到的照片空间上比较集中
   - 匹配到的照片时间上形成合理聚类
7. 后端保存地点坐标和照片绑定结果。

如果没有可靠匹配结果，仍然允许保存该地点，并将状态标记为 `unmatched`。

前端可以简单提示“暂时没有找到照片”，但不能阻塞用户继续搭建路线。

## 腾讯地图使用方式

正式产品后端使用腾讯地图 WebService API，不依赖 MCP。

腾讯地图 MCP 只作为开发阶段测试工具，方便在 Codex 中快速验证地点搜索结果。

行政区划数据：

- 使用 `GET /ws/district/v1/list`
- 使用 `struct_type=1` 拉取省、市、区县嵌套数据
- 后端将行政区划数据存入数据库

地点解析：

- 用户输入二级地点时，调用腾讯地图地点提示或 POI 搜索
- 查询时带上当前城市上下文
- 返回候选后，再按省份、城市、POI 类型和照片 GPS 匹配结果做筛选

## Photo-Agent 边界

Photo-Agent 不关心路线、城市、地点或文章。

Photo-Agent 只负责：

- 扫描本地照片目录
- 读取照片 EXIF GPS
- 读取照片拍摄时间
- 读取照片相对路径
- 生成缩略图和预览图
- 上传照片索引数据
- 按需要上传缩略图或预览图

云端后端负责：

- 路线结构
- 城市选择
- 地点解析
- GPS 照片匹配
- AI 写作上下文组织

## 产品原则

老人端正常操作流程必须简单：

```text
选城市 -> 输入地点 -> 系统找照片 -> 保存
```

系统应该让后端多做一点智能判断，少让用户在前端多做选择。
