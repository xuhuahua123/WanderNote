# 004. AI 写作服务决策记录

## 状态

已确认。

## 背景

前端已经确认 AI 写作流程：

- 写某个地点
- 写整条路线
- 重新生成
- 更朴实一点
- 更有感情一点

后端需要统一封装 AI 写作能力，避免前端直接调用模型接口。

MVP 主模型使用小米 MiMo-V2.5 API。

## 核心决策

AI 写作由云端后端负责。

前端只提交写作对象、已选照片、情绪选项和补充说明。

后端负责：

- 组装路线/地点上下文
- 组装照片上下文
- 构造提示词
- 调用 MiMo-V2.5 API
- 解析结构化结果
- 保存当前草稿
- 记录 AI 调用日志
- 返回文章结果

## 模型配置

MVP 使用：

```text
AI_PROVIDER=mimo
AI_MODEL=mimo-v2.5
AI_API_KEY=xxx
```

后端封装统一服务：

```text
AiWritingService
```

后续如果切换 GLM、Kimi、DeepSeek 或其他模型，不影响前端接口。

## 支持动作

AI 写作服务支持四类动作：

```text
generate
regenerate
make_plainer
make_more_emotional
```

对应前端按钮：

- 生成游记
- 重新生成
- 更朴实一点
- 更有感情一点

## 写某个地点

输入上下文：

- 路线名称
- 年份
- 省份
- 城市/地区
- 地点名称
- 已选照片列表
- 照片拍摄时间
- 照片对应的地点信息
- 用户选择的情绪
- 用户补充说明，可为空

生成目标：

- 围绕当前地点写一篇公众号游记
- 重点写这一站的风景、路上生活、心情和感受
- 为每张选中照片生成图片说明

## 写整条路线

输入上下文：

- 路线名称
- 年份
- 城市顺序
- 地点顺序
- 已选代表照片
- 每张照片所属城市/地点
- 用户选择的情绪
- 用户补充说明，可为空

生成目标：

- 按路线顺序写整趟旅行
- 写出从出发到返程的线路感
- 体现退休后自驾、路上吃住、准备充分、辛苦但值得的生活感
- 结尾写回忆和感悟
- 为每张代表照片生成图片说明

## 输出格式

模型输出必须尽量要求为 JSON。

推荐结构：

```json
{
  "title": "退休后第一次近看珠峰，心里满是震撼",
  "content": "正文内容...",
  "imageCaptions": [
    {
      "photoId": "p001",
      "caption": "蓝天下的珠峰，远远望去格外清晰。"
    }
  ]
}
```

后端需要做容错：

- JSON 解析失败时，尝试提取标题和正文。
- 图片说明缺失时，后端可填空字符串。
- 返回内容为空时，提示前端重试。

## 文风要求

主风格：

```text
退休自驾游记，温暖、朴实、有生活气，适合公众号发布。
```

必须避免：

- 写成旅游攻略
- 写成广告文案
- 写成营销号标题
- 过度夸张
- 过度年轻化
- 堆砌成语
- 编造用户没有提供的具体事实

重点鼓励：

- 写退休后的自由感
- 写自驾长线的成就感
- 写路上的辛苦和踏实
- 写车上吃住和充分准备
- 写家人或同行者的互相照应
- 写雪山、草原、边疆、古城带来的震撼
- 写朴实生活里的感受

## 更朴实一点

基于当前文章改写。

目标：

- 减少夸张表达
- 减少华丽修辞
- 更像真实生活记录
- 保留原有事实和照片说明

不重新选图。

不重新进入写作准备页。

## 更有感情一点

基于当前文章改写。

目标：

- 增加回忆感
- 增加旅途情绪
- 增加退休后自驾的感悟
- 保持自然，不煽情过度

不重新选图。

不重新进入写作准备页。

## 重新生成

重新生成规则：

- 保留当前照片。
- 保留当前情绪选项。
- 保留用户补充说明。
- 生成一篇新文章。
- 替换当前草稿。
- 前端必须二次确认。

## 草稿保存

AI 返回成功后，后端写入或更新 `article_draft`。

每个写作对象只保留一份当前草稿：

```text
target_type = route / place
target_id = route.id / route_place.id
```

不做草稿历史。

## 调用日志

每次调用 AI 都写入 `article_generation_log`。

记录：

- 写作对象
- 动作类型
- 模型供应商
- 模型名称
- 请求内容
- 返回内容
- token 数
- 成本
- 是否成功
- 错误信息

该表只用于开发排查、成本统计和提示词优化。

不在前端展示为草稿历史。

## 成本控制

MVP 成本控制规则：

- 一篇文章最多使用 10 张照片。
- 正文建议 1200 到 1800 字。
- 记录 token 和调用成本。
- 失败时提示用户重试。
- 不做高并发队列。
- 不让前端直接调用模型 API。

## 后端接口方向

```text
POST /api/writing/generate
POST /api/writing/regenerate
POST /api/writing/make-plainer
POST /api/writing/make-more-emotional
PUT  /api/article-drafts/{id}
```

接口细节后续在接口协议文档中确认。

## 产品原则

AI 是帮用户把旅行经历写出来，不是替用户编故事。

核心原则：

```text
不编造事实。
不过度营销。
不复杂配置。
以退休自驾生活感为主。
输出结构稳定，方便前端编辑和复制。
```
